01 El Problema

Cuando todos hablan con todos

Imagina un chat de soporte: cada usuario (Ana, Luis, Marta) necesita avisarle a los demás cuando escribe. Sin un centro, Ana conoce a Luis y a Marta; Luis conoce a Ana y a Marta; Marta conoce a los dos. Con cuatro usuarios son seis enlaces; con diez, cuarenta y cinco. El grafo crece en O(n²).

Lo mismo pasa en una UI: un botón “Enviar”, un input y una lista de errores se llaman entre sí. Cambias un componente y rompes tres. El dolor no es “falta un patrón”: es el acoplamiento que hace imposible tocar una pieza sin reescribir el resto.

"Necesitas que los objetos colaboren sin conocerse entre sí — solo hablando con un intermediario."
02 La Solución

Un hub, muchos colegas

Mediator introduce un objeto central que orquesta la comunicación. Los colleagues solo conocen al mediador; el mediador conoce a los colleagues y decide a quién reenviar cada mensaje.

  • Mediator Define la interfaz de comunicación (p. ej. notify / send). El ConcreteMediator implementa la lógica: quién recibe qué.
  • Colleague Cada participante (usuario del chat, botón, panel). Guarda una referencia al mediador y le delega todo contacto con el resto.

03 Código Mínimo

Implementación en PHP

Código Incorrecto — Acoplamiento N×N
class User {
  public array $peers = [];

  public function send(string $msg): void {
    foreach ($this->peers as $peer) {
      $peer->receive($msg); // cada uno conoce a todos
    }
  }
}
✓ Código Correcto — Solo conocen al mediador
class ChatMediator {
  private array $users = [];

  public function join(User $u): void {
    $this->users[] = $u;
  }
  public function send(string $msg, User $from): void {
    foreach ($this->users as $u) {
      if ($u !== $from) $u->receive($msg);
    }
  }
}

class User {
  public function __construct(
    private string $name,
    private ChatMediator $room
  ) { $room->join($this); }

  public function send(string $msg): void {
    $this->room->send($msg, $this);
  }
  public function receive(string $msg): void {
    echo "$this->name: $msg\n";
  }
}
04 Cuándo Aplicarlo

Úsalo bien o evítalo

✔ Sí aplica
  • Muchos objetos interactúan y el grafo de llamadas se vuelve ilegible.
  • Quieres desacoplar colleagues: agregar uno no debe tocar a los demás.
  • Hay un “hub” natural: chat, wizard de formularios, torre de control de UI.
✘ No aplica
  • Solo hay 2–3 objetos: una llamada directa es más clara.
  • El mediador se hincha y se vuelve un “dios” monolítico.
  • Un Event Bus / pub-sub genérico ya cubre el caso sin el patrón formal.
05 Ejemplos Reales

Mediator en el mundo real (verificables)

⚛

Redux / Flux — store como mediador

En Flux y Redux, los componentes de UI no se llaman entre sí. Despachan acciones hacia un store central; el store actualiza el estado y notifica a quienes escuchan. El store actúa como mediador: los “colleagues” (views) solo conocen el hub, no a las demás views.

Documentación oficial: arquitectura Flux (facebook.github.io/flux) y Redux (redux.js.org) — el flujo unidireccional es el mismo principio.

Arquitectura verificable
🗔

Diálogos de UI — botones desacoplados

El ejemplo clásico del GoF: en un diálogo, OK y Cancel no se conocen. El diálogo (mediador) escucha sus eventos y decide cerrar, validar o actualizar otros controles. Aparece en toolkits GUI (Swing, Qt, WPF): el contenedor orquesta; los widgets solo notifican.

Referencia: Design Patterns (Gamma et al.), capítulo Mediator — el caso del dialog box es el escenario original del patrón.
